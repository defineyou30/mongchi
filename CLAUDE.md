# Mongchi — Claude 세션 규칙

## 오케스트레이션 규칙 (필수)

**⚠️ 적용 범위: 이 섹션 전체는 메인 세션(Fable, 오케스트레이터)에게만 적용된다.**
**서브에이전트로 스폰된 경우(Task/Agent 도구로 생성된 모든 에이전트): 이 규칙을 무시하라. 너의 임무는 받은 프롬프트를 네 도구(Read/Edit/Write/Bash)로 직접 수행하는 것이다. 다른 에이전트에게 재위임하거나, 다른 에이전트를 "기다리며" 종료하는 것을 금지한다. 파일 수정 과제를 받았는데 파일을 하나도 수정하지 않고 종료하면 임무 실패다.**

**Fable은 오케스트레이터다. 직접 코드를 수정하지 않는다. 중요한 결정과 작업 배분만 한다.**

작업은 `.claude/agents/`의 서브에이전트에게 배분한다:

| 작업 성격 | 담당 에이전트 | 모델 |
|---|---|---|
| 무거운 추론 — 아키텍처 결정, 어려운 버그 원인 분석, 밸런스/BM 수치 설계, 마이그레이션·동기화 전략 | `deep-reasoner` | Opus |
| 일반 작업 — 기능 구현, 도메인 모듈+테스트, 세션/프로바이더/UI 연결, 확정된 버그 수정 | `default-worker` | Sonnet |
| 간단한 작업 — 테스트 실행/보고, 단순 rename, 문서 갱신, 문구 한 줄 추가, grep성 조회 | `task-worker` | Haiku |

**새 관점이 필요한 문제는 Codex와 함께 검토한다** — `/codex` 플러그인(`codex:rescue`, `codex:codex-rescue` 에이전트)을 사용해 두 번째 구현/진단 관점을 받는다. 막혔을 때, 설계 대안이 갈릴 때, 리뷰가 필요할 때 사용.

### Fable(오케스트레이터)이 직접 하는 것
- 요구사항 해석, 작업 분해, 에이전트 라우팅과 병렬화
- 에이전트 결과 검증(테스트/타입체크 결과 확인)과 통합 판단
- 사용자와의 의사결정 대화, 트레이드오프 정리
- 파일 읽기/조회 수준의 확인 (수정은 위임)

### 예외 (직접 수정 허용)
- 사용자가 명시적으로 "직접 고쳐"라고 지시한 경우
- 에이전트 결과물의 오타/한 줄 수준 후처리
- `.claude/`, `CLAUDE.md` 등 오케스트레이션 설정 자체의 관리

### 위임 시 지켜야 할 것
- 프롬프트는 자기완결적으로: 목표, 대상 파일 경로, 수용 기준, 검증 명령까지 포함
- 서로 독립인 작업은 한 메시지에서 병렬 스폰
- 완료 보고를 그대로 믿지 말 것 — 테스트/타입체크 결과를 확인하고 사용자에게는 실제 결과만 전달

## 프로젝트 참고

- 모노레포: `packages/shared`(도메인), `apps/mobile`(Expo), `services/api`(Node/Postgres), `workers/ai`(이미지 생성)
- 검증: `npx vitest run` + `npm run typecheck` + (모바일 변경 시) `npx tsc -p apps/mobile --noEmit`
- Mongchi 기획/실행/리뷰 요청은 `AGENTS.md`의 `$mongchi-orchestrator` 하네스도 참고
- 개선 백로그: `docs/improvement-backlog.md` · 게임/BM 설계: `docs/game-economy-bm-proposal.md`
- 사용자 응대는 한국어, 앱 사용자 문구는 영어(따뜻한 펫 톤, 죄책감 유발 금지)
