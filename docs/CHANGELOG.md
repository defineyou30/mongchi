## 1. 개요 (무엇을·왜)

2026-07-07, Mongchi에 처음으로 git 버전관리가 도입됐다(그 전까지 버전관리 0 — 출시 준비 감사가 꼽은 최상위 리스크). 초기 커밋 이후 이틀 동안 20개 커밋이 랜드됐고, 각 커밋은 통합검증(vitest + 타입체크)을 그린으로 통과한 뒤에만 들어갔다. 이 기간의 작업은 세 축으로 나뉜다. (1) **출시 위생·안전** — 보안 봉합, 법적 문서, 관측성, 데이터 백업으로 "실제로 스토어에 낼 수 있는 상태" 만들기. (2) **게임필·사운드** — "부재 중 사건 0"·"다 해줬는데 게이지 안 오름" 같은 리텐션 급소를 케어 순간 연출·세계 자율성·아이템 개별성·산책 대기 경험·BGM/앰비언스로 메우기. (3) **경제·구조 토대** — 서버 크레딧 원장으로 로컬 변조 무한 무료생성 구멍을 봉합하고, 멀티펫(슬롯)의 도메인·서버 네임스페이스를 미리 깔아 v1.1 확장 여지를 확보. vitest는 세션 시작 시 ~730개에서 1252개 그린으로 늘었고, 세션 스키마는 v7(pets 번들)로, Supabase 마이그레이션은 0001–0005까지 배포됐다.

## 3. 이 기간의 핵심 결정

### 3.1 슬롯(멀티펫)은 출시 후 v1.1로 연기
출시 전엔 "한 마리라도 되는가"(핵심 훅)를 먼저 검증한다. 실유저 0명 상태에서 D30 리텐션 이론논쟁은 지양. 재사용 가능한 토대(W1 도메인·W2 서버)는 랜드·배포했고, W3(구매+2번째 펫 온보딩)는 `git stash 'slot-w3-wip'`로 보관, W4(멀티펫 표면)는 미착수. 근거와 상세는 [`multi-pet-slot-plan.md`](./multi-pet-slot-plan.md).

### 3.2 크레딧 IAP는 RevenueCat 사용
영수증 검증·환불 웹훅·크로스플랫폼을 RevenueCat이 대신하므로 직접 `verify-purchase` 구현이 불필요(이게 결제 검증 위험의 핵심이었음). 셋업 무관 코드(크레딧 팩 카탈로그·상점 UI·grant 웹훅 Edge)는 후속 구축하고, RevenueCat 연결은 "결제 켜기" 최종 단계로 남긴다. 스토어 상품 생성·RC 대시보드 매핑은 출시 직전 사장 액션.

### 3.3 크레딧 경제 데드엔드 인지
현 코드엔 크레딧 구매 SKU가 없고(구독 `premium_chat_monthly`만), 반복 크레딧 파우셋(데일리)도 없다. 소비처(표정팩·재생성)는 있는데 버는 곳(무료 60개 소진 후 0)·사는 곳이 둘 다 없어 경제가 데드엔드 상태다. → 출시 전 매출 급소로 명시했고, RevenueCat 토대가 처방. 서버 원장(1a–1c)·안전(c5d405c)은 이미 봉합해 두어 결제만 얹으면 되는 구조.

### 3.4 분석 스냅샷 기반 개발
7/4 리텐션 갭 분석, 7/7 출시준비 진단에서 지적된 급소를 이 기간 작업이 직접 겨냥했다. 날씨 시드 고정 버그·알림 파이프라인 데드코드·"부재 중 사건 0"·케어 밸런스가 각각 수정/Tier 2–4/밸런스 조정으로 해소됐다. 분석 스냅샷 문서는 본문을 보존하고 하단에 해결 현황을 덧붙이는 방식으로 갱신한다.

## 4. 출시 전 남은 것

- **크레딧 IAP SKU** — RevenueCat 배선 + 사장 스토어 셋업 (매출 급소)
- **반복 크레딧 파우셋(데일리)** — 여전히 없음, 검토 대상
- **delete-account Edge Function** — 서버 데이터 완전 삭제(Apple 계정삭제 요건·GDPR), 진행 중, 출시 필수
- **Sentry 실연동** — 후속 네이티브 재빌드 묶음
- **dev/prod Supabase 분리**, **free_limit 복원**, **익명 로그인 rate-limit/CAPTCHA**(사장 대시보드)
- **산책 도감 9종 OS 이모지 → 픽셀 아이콘** — 미착수
- **cleanliness HUD 미노출** — Bath 액션은 뚫렸으나 미터 표시는 별도
