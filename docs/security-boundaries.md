# Security And Provider Boundaries

This scaffold intentionally does not connect production services.

## Mobile App Must Not Contain

- AI provider keys.
- Service role keys.
- Payment verification secrets.
- Public original-photo URLs.
- Raw production storage credentials.
- Exact location history or weather provider credentials.

- Purchase verification and entitlement ledger. Initial tested commerce ledger supports mock server verification, server-side store verifier injection, a tested HTTP store verification gateway, tested direct App Store Server API and Google Play Developer API verifier adapters with explicit Google Play subscription product ids in production config, request-scoped raw store verification token validation for purchase and restore verifier use, idempotent grants, restore, transaction reuse protection, secret-guarded server-only revocation webhooks, and a tested `/v1/commerce/store-webhooks` ingress that normalizes App Store Server Notification v2 refund/revoke payloads to transaction revokes, verifies direct App Store notification/transaction JWS payloads against the `x5c certificate chain` and trusted SHA-256 root certificate fingerprint, and maps Google Play RTDN revocations to receipt-hash revokes without persisting raw purchase tokens; the Postgres-backed request service now covers entitlement and purchase-ledger state without persisting raw store tokens and enqueues sanitized `commerce.purchase_revoked` outbox audit events with hashed aggregate ids. Store account/product setup, deployed store webhook registration, and monitoring still need to be connected.
- Mock purchase verification is local-only. Production release validation requires `TINY_PET_API_ALLOW_MOCK_PURCHASES=false`, and service/router/server mounts can set `allowMockPurchaseVerification: false` so mock receipt grants and restore calls return `purchase_verification_unavailable`.
- Store receipt verification adapters belong in the API server only. The tested `StorePurchaseVerifier` injection point, HTTP verifier adapter, and direct Apple/Google adapters can receive request-scoped purchase or restore `storeVerificationToken` values and must return only normalized server-side results; direct App Store runtime config also verifies server-returned `signedTransactionInfo` JWS payloads before purchase grants. Mobile code must never receive provider credentials, payment secrets, or raw provider responses, and analytics must not contain receipt or token payloads. `npm run validate:mobile-secret-boundaries` now statically fails if server-only `TINY_PET_*` env keys, unsupported mobile public env keys, secret-shaped public env names, or real-looking provider/storage/payment secrets appear in the mobile app surface.
- Private storage credentials belong only in backend API/worker processes. The tested API `PrivateStorageSigner` injection point, S3-compatible signing adapter, S3-compatible object deleter, and worker S3 read/write adapter keep upload/read/delete/write credentials server-side; mobile code must never contain storage credentials, bucket service keys, or provider signing material.

A tested outbox worker process runnable with `npm run start:outbox-worker` can deliver those sanitized audit events through an injected sink or the safe operational logger and mark processed/failed delivery state. Production scheduler infrastructure and alert routing still need to be connected.

## AI Worker Future Ownership

- Photo validation and EXIF stripping. Initial tested worker intake now checks JPEG/PNG/WebP signatures, parses and limits header dimensions, verifies JPEG terminal EOI and scan payload structure, PNG chunk CRC/IDAT/IEND structure, inflates PNG IDAT data to validate raster scanline lengths and filter bytes, verifies WebP RIFF chunk bounds and image chunk presence, runs Sharp-backed full pixel decode for JPEG/PNG/WebP source photos before provider input, rejects content-type mismatch, and strips JPEG APP1 EXIF metadata.
- Upload safety precheck. The worker runtime now has a tested source-photo safety checker contract that runs before provider generation, can wrap an injected provider/CV classifier, fails closed when the classifier is unavailable, preserves unsafe/manual-review failure metadata, and includes a tested OpenAI Responses vision classifier for production mounting with worker-only credentials.
- Provider calls for avatar generation. A tested OpenAI image edit adapter exists for `/images/edits`; production deployment still needs worker-only API key configuration, model selection, retry/cost monitoring, and scheduler infrastructure mounting.
- Postprocess, thumbnails, and provider-backed quality signal extraction. A structured quality gate and tested OpenAI generation quality evaluator exist for required asset states, species match, one-pet visibility signals, safety approval, style match, provider confidence, runtime-configurable thresholds, and manual review routing; the worker runtime can now claim jobs through a tested batch runner, invoke injected provider and tested S3-compatible storage adapters, and persist completed or failed state without exposing provider outputs to mobile.
- Cost and failure metadata. Worker and process runners now return safe status/failure metadata and persist coarse failure codes/messages without exposing source-photo bytes, provider payloads, raw provider responses, storage credentials, or private storage URIs to mobile; the default runtime logger now emits sanitized `operational_alert_triggered` events for cost spikes and failure categories, including commerce store webhook rejection, and can fan those alerts out to a tested sanitized webhook sink without blocking runtime behavior. Production still needs the chosen alert route deployed.

## Current Mock Policy

- Free reactions are selected locally from authored rules.
- Mock generation uses static metadata and mock signed read URLs only; mobile falls back to bundled pet art for those non-renderable local URLs.
- Mock photo upload never sends files to a provider or storage service; API service tests verify type/size/hash metadata and ownership before allowing generation-job creation.
- Shop can read server-owned product and entitlement state in API-backed integration builds. Native checkout is available only in custom builds with `EXPO_PUBLIC_TINY_PET_ENABLE_NATIVE_CHECKOUT=true`, and purchase grants still require server verification before entitlement state updates.
- Premium chat stays disabled in local prototype mode. API-backed integration builds use server entitlement checks and can run either the backend-only local chat provider or the injected OpenAI provider; mobile never receives provider keys or raw provider responses.
- Development mock auth fallback is only for local adapter builds. Production release validation requires it to be disabled and rejects public mock auth tokens.
- API mock auth headers are also local-only. Production release validation requires `TINY_PET_API_ALLOW_MOCK_AUTH=false`, and server mounts can set `allowMockAuth: false` until real auth middleware is connected.
- Auth provider verification belongs in the API server only. The tested `ApiSessionVerifier` and JWT/JWKS adapter accept bearer tokens from mobile and return normalized user identity after issuer, audience, expiry, not-before, key id, and signature checks; mobile code must never contain provider secrets, service role keys, or private verification material.
- API mock purchase verification is also local-only. Production release validation requires `TINY_PET_API_ALLOW_MOCK_PURCHASES=false` plus either the HTTP verifier gateway env (`TINY_PET_STORE_VERIFIER_ENDPOINT`, `TINY_PET_STORE_VERIFIER_API_KEY`) or direct verifier env (`TINY_PET_STORE_VERIFIER_PROVIDER=direct` with Apple/Google server credentials and `TINY_PET_APP_STORE_NOTIFICATION_ROOT_CERT_SHA256`), and server mounts can set `allowMockPurchaseVerification: false` with server-side store verification injected.
- Store refund/revocation webhooks must be server-to-server only. The tested routes require `x-tiny-pet-commerce-webhook-secret` to match server config and are not available through the mobile API client. `/v1/commerce/store-webhooks` accepts raw App Store Server Notification v2 and Google Play RTDN payloads, structurally validates App Store compact JWS payloads by requiring supported ES256 protected headers plus payload/signature segments before decoding, verifies the direct App Store `x5c certificate chain` and ES256 signature when runtime config provides `TINY_PET_APP_STORE_NOTIFICATION_ROOT_CERT_SHA256`, checks configured App Store bundle id / Google Play package name when available, ignores non-revocation notification types with a 202 response, emits safe processed/ignored/rejected operational events without transaction ids, receipt hashes, purchase tokens, or raw payloads, revokes Google Play purchases by hashing the RTDN purchase token before ledger lookup so the raw token is not stored, and records sanitized `commerce.purchase_revoked` outbox audit events without raw store identifiers.
- API mock storage signing is also local-only. Production release validation requires `TINY_PET_API_ALLOW_MOCK_STORAGE=false` plus `TINY_PET_STORAGE_BUCKET`, `TINY_PET_STORAGE_REGION`, `TINY_PET_STORAGE_ACCESS_KEY_ID`, and `TINY_PET_STORAGE_SECRET_ACCESS_KEY`, and server mounts can set `allowMockStorageSigning: false` with the S3-compatible signer injected from server-side credentials.
- The JSON snapshot store is local/integration infrastructure only and must not be treated as production storage for private photos, generated assets, purchase receipts, or chat retention.
- The snapshot-to-Postgres repository is for migration rehearsal and integration QA only; production reads/writes should be owned by the request-scoped repository bundle behind production auth.
